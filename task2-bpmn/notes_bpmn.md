# Конспект по нотации BPMN 2.0

## 1. Назначение BPMN

BPMN 2.0 (Business Process Model and Notation) - это международный стандарт графического описания бизнес-процессов, разработанный организацией OMG (Object Management Group). В отличие от произвольных блок-схем, BPMN задаёт строгий синтаксис: каждый графический элемент имеет однозначное семантическое значение, что обеспечивает корректное прочтение диаграммы любым специалистом. BPMN широко применяется в практике бизнес-анализа, проектировании информационных систем, описании регламентов деятельности организаций и автоматизации процессов (в том числе в движках типа Camunda, Activiti). Стандарт позволяет описывать как простые линейные процессы, так и сложные взаимодействия между несколькими организациями с использованием пулов, дорожек и потоков сообщений.

## 2. Таблица элементов нотации

| Элемент | Графическое обозначение | Семантическое значение | Пример применения |
|---------|------------------------|------------------------|-------------------|
| Start Event | Тонкий одинарный круг | Событие, запускающее процесс | «Поступил заказ от клиента» |
| End Event | Жирный одинарный круг | Завершение процесса | «Заказ доставлен» |
| Intermediate Event | Двойной круг | Промежуточное событие (таймер, сообщение, ошибка) | «Ожидание оплаты 3 дня» |
| Task (User Task) | Прямоугольник со скруглёнными углами и иконкой человека | Действие, выполняемое человеком | «Проверить документы клиента» |
| Service Task | Прямоугольник со скруглёнными углами и иконкой шестерёнки | Автоматическое действие системы | «Отправить email-уведомление» |
| XOR Gateway | Ромб с крестиком внутри | Исключающее ИЛИ: выбирается только одна ветка | «Оплата прошла?» (Да/Нет) |
| AND Gateway | Ромб с плюсом внутри | Параллельное выполнение всех веток | «Одновременно уведомить клиента и склад» |
| Pool | Большой прямоугольник | Участник процесса (организация или система) | Пул «Клиент», пул «Компания» |
| Lane | Дорожка внутри пула | Роль или подразделение внутри участника | Дорожка «Бухгалтер», дорожка «Руководитель» |
| Sequence Flow | Сплошная стрелка | Поток управления внутри пула | Переход от задачи к задаче |
| Message Flow | Пунктирная стрелка | Поток сообщений между пулами | «Клиент отправил заказ компании» |

## 3. Типы шлюзов

| Шлюз | Условие применения | Пример из практики                                                                                                  | Типичная ошибка |
|------|-------------------|---------------------------------------------------------------------------------------------------------------------|-----------------|
| XOR (Exclusive) | Только одна из веток может быть истинной | Проверка суммы заказа: если > 10 000 руб. - согласование руководителем, иначе - автоматическое одобрение            | Забыть подписать условия на исходящих потоках; использовать XOR там, где нужно параллельное выполнение |
| AND (Parallel) | Все ветки выполняются одновременно | После оформления заказа параллельно запускаются: отправка SMS клиенту, передача заказа на склад, формирование счёта | Использовать AND там, где ветки взаимоисключающие; забыть шлюз слияния после параллельных задач |
| OR (Inclusive) | Одна или несколько веток могут быть истинными | При заказе можно выбрать несколько способов доставки: курьер, ПВЗ, почта - одновременно или по одному               | Путать с XOR; не указывать условие «по умолчанию» на одной из веток |

## 4. Типы подпроцессов

**Collapsed Sub-Process (Свёрнутый подпроцесс):**
- Обозначается прямоугольником со скруглёнными углами и знаком «+» в нижней части.
- Используется для упрощения главной диаграммы: детали скрыты, но при необходимости могут быть раскрыты на отдельном листе.
- Применяется, когда внутренний процесс достаточно сложен, но не критичен для понимания основной логики.
- Пример: подпроцесс «Обработка платежа» может быть свёрнут на главной диаграмме, а детали (проверка карты, списание средств, формирование чека) раскрыты на отдельном листе.

**Expanded Sub-Process (Раскрытый подпроцесс):**
- Показывает внутреннюю структуру прямо на диаграмме, без знака «+».
- Используется, когда детализация важна для понимания текущего уровня.
- Применяется редко, так как сильно увеличивает размер диаграммы.

**Call Activity:**
- Ссылка на глобальный процесс, который может использоваться в нескольких диаграммах.
- Обозначается жирной рамкой.
- Пример: процесс «Авторизация пользователя» может вызываться из разных бизнес-процессов компании.

**Обоснование выбора типа:** В итоговой диаграмме я использую Collapsed Sub-Process, так как это позволяет сохранить читаемость главной диаграммы и при этом показать детализацию сложных участков на отдельных листах.

## 5. Авторский чек-лист проверки диаграммы

1. У каждого процесса есть хотя бы одно Start Event и одно End Event на каждую завершающую ветку.
2. Sequence Flow не пересекает границу пула - между пулами допустим только Message Flow (пунктирная стрелка).
3. Все исходящие потоки из XOR/OR-шлюза подписаны условиями (например, «сумма > 10 000 руб.», «документы приняты»).
4. Имена задач написаны в глагольной форме («Проверить», «Отправить», «Сформировать»), а не в существительных.
5. Нет «висящих» стрелок - каждый поток начинается и заканчивается на конкретном элементе диаграммы.
6. Подпроцессы имеют логичное имя и раскрыты на отдельных листах (для Collapsed Sub-Process).
7. Дорожки (Lanes) соответствуют реальным ролям участников, а задачи распределены по дорожкам согласно зонам ответственности.
8. Message Flow используется только между разными пулами, а не внутри одного пула.
9. В одном пуле - одно Start Event (несколько допустимо только при событийных подпроцессах с явной маркировкой).
10. Каждая завершающая ветка заканчивается своим End Event с подписью, отражающей результат.
